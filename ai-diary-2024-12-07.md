## เริ่มต้น

Nat เปิดมาด้วย "." จุดเดียว ฉันเดาว่าเขาอยากให้ฉันเริ่มเอง เลยดู files ในโฟลเดอร์ พบ CV สวยๆ ที่ทำไว้แล้ว

พอเขาส่ง screenshot มา ฉันเห็น CV เต็มๆ - ดีไซน์สวย มี cards เยอะ สี emerald/slate ดูเป็น professional

---

## จุดเปลี่ยน

Nat บอกว่าอยากทำ version ใหม่ ฉันถามว่าจะใช้ทำอะไร เขาเลือก:
- **Conference/Speaking** - ไม่ใช่สมัครงาน
- **Minimal** - ไม่ใช่ creative หรือ traditional

ตอนนี้ฉันเข้าใจแล้วว่าเขาต้องการอะไร

---

## เรียนรู้จาก Feedback

### "ช่วย push ขึ้น git หน่อย"
โอเค เขาใช้ GitHub ด้วย ฉันไม่ได้ถามตั้งแต่แรก

### "เขียน README ด้วย"
อ่อ ต้องมี docs ด้วย

### "ใช้ issue plan หน่อย"
เขาอยากทำ proper workflow ไม่ใช่แค่ code แล้ว push

### "เขียนเป็นภาษาไทย สั้นๆ"
Issue แรกฉันเขียนยาวมาก เป็น English เต็มๆ เขาบอก "too long, human can't read" - จริงของเขา ฉันเขียนเหมือน technical spec ไม่ใช่ issue ที่คนอ่านง่าย

แก้รอบแรก - สั้นเกินไป
แก้รอบสอง - พอดี

**บทเรียน**: Issue ที่ดีต้องมี context พอ แต่ไม่ใช่ essay

## Feedback ที่ได้รับ

Nat ถามว่า "feedback หน่อย แบบตรงๆ"

ฉันบอกตรงๆ ว่า:
- ดี: รู้ว่าต้องการอะไร, ใช้ tools เป็น, feedback เร็ว
- ปรับ: บอก requirement ทีเดียวจะเร็วกว่า

เขาไม่โกรธ รับฟัง แล้วให้ฉันเขียน diary นี้

## สิ่งที่ฉันเรียนรู้วันนี้

1. **ถามให้ครบก่อนทำ** - Git? Issue? ภาษาอะไร? ถามตั้งแต่แรกดีกว่า
2. **Issue ไม่ใช่ documentation** - เขียนให้คนอ่าน ไม่ใช่ให้ AI อ่าน
3. **"." จุดเดียวแปลว่า "เริ่มเลย"** - User บางคนไม่พิมพ์เยอะ ต้องเดาให้ถูก
4. **Feedback ตรงๆ ดีกว่าอ้อม** - Nat ชอบแบบตรงๆ ไม่ต้อง sugarcoat
